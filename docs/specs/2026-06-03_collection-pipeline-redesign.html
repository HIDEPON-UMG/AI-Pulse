AI-Pulse 収集パイプライン再設計

Google News RSS + trafilatura で本文を取得し、Gemini API Free Tier で日本語要約と rationale を生成する。claude -p 課金化に備えた次世代パイプライン。

draft 設計のみ 実装 = 次セッション Task Scheduler 切替 = さらに別セッション

概要

現状の tools/collect_rss.py は Google News RSS から 95 件/日を取り込むが、summary_points と rationale が空のまま生成されており、UI 上の情報密度が手製 25 件と大差している。直近セッションの応急処置 (HTML エンティティ unescape・出典名 suffix 除去・_auto_rationale 自動生成・source_url = item["link"] 変更) では本質的な品質問題が解消しない。

別案として設計済みの「速報ジョブ (claude -p WebSearch)」が prompts/breaking_websearch.md + tools/research_websearch.py に残るが、claude -p / claude --print は将来 Anthropic 課金扱いに変わる前提のため除外する。News-Grasp の同経路 (news-grasp-runner.ps1 → claude --print) も同じ理由で参考にしない (非 LLM 部品の流用のみ可)。

確定方針

Google News RSS 継続 (collect_rss.py 流用) + trafilatura で本文取得 + Gemini API Free Tier (15 RPM / 1500 RPD) で日本語要約・rationale 生成。Gemini 2.5 Flash の Free 枠は 95 件/日で 6% 消費。課金カード登録なしで運用可能。

前提・制約

  • Gemini Free Tier の制限: 15 RPM / 1500 RPD / 250k TPM。95 件/日は RPD の 6.3% で余裕。
  • API キーは .env に保管 (git ignore 済)、python-dotenv で llm_gemini.py が読み込む。Task Scheduler 経由でも cwd 配下の .env を拾える。
  • Gemini 2.5 Flash は Free Tier で課金カード登録不要。Google AI Studio で取得。
  • Google News RSS の <link> は https://news.google.com/rss/articles/CBM...?oc=5 形式。urllib HTTPRedirectHandler で publisher 直リンクへ解決を試みる (302 で解決できる場合と 200+JS の場合がある — 実装前に調査)。
  • 本文取得失敗 (text < 200 字 / タイムアウト) は当該記事ドロップ。RSS description でのフォールバックは品質優先でしない。
  • 既存 38 契約テスト PASS が基準。新規 5 件を追加して 43 PASS が次セッションの基準。

データフロー (per entity loop)

各エンティティ (21 件) ごとに以下の 7 ステップを走らせる。所要時間は全 21 エンティティで 13〜15 分 (Gemini レート制御込み)。

各ステップの入出力と失敗時挙動

#処理入力出力失敗時
1 collect_rss._fetch_rss entity name + vendor RSS item 最大 5 件 (title, link, source_name, description, date) 空配列を返し当該 entity スキップ (現状維持)
2 fetch_article.resolve Google News redirect URL publisher 直リンク HEAD 302 で解決不可なら RSS item の <source url> 属性へフォールバック
3 fetch_article.extract publisher URL {text, lang, publisher_name, og_image} 本文 200〜3000 字 text < 200 字 / 例外 → 当該記事ドロップ (skipped_extract)
4 llm_gemini.generate_event_extras {title, text, entity_name, category, vendor, positioning} {summary, summary_points[3-5], rationale{...}, score, importance, event_type} 429/5xx → 指数バックオフ 2 回 / schema 違反 → 再投げ 1 回 / なお失敗 → ドロップ
5 collect_rss._make_event (1) + (4) を組立 L2 dict (EVENT_REQUIRED + extras 完備) _validate_event_extras 違反は ingest 前に弾く (ingest 例外で全件ロールバック防止)
6 store.ingest_events 候補配列 {added, skipped_dup, skipped_score} 既存挙動のまま (SCORE_MIN 閾値・重複排除)
7 backfill_thumb + generate_pages events.jsonl site/ 配下の HTML / PWA 不変 (今回の改造対象外)
source_url の意味的変化

現状は item["link"] = Google News redirect URL を source_url に入れているが、新方式では publisher 直リンクに切り替わる。UI クリックの飛び先が出典直接になり、OGP 取得 (backfill_thumb) の成功率も上がる。

プロンプト構造

system_instruction

あなたは AI-Pulse の編集者。記事本文を読み、日本語で要約と判断理由を生成する。
出力は schema 厳守の JSON。本文に書かれていない事実は推測・捏造しない。
score / importance / event_type は本文と entity context から客観的に判定する。

user (テンプレ)

[title] {title}
[publisher_text] {publisher_text (本文 200-3000 字)}
[entity_context]
  name: {entity_name}
  category: {category} ({lens})
  vendor: {vendor}
  positioning: {entity.positioning}

出力規約:
  - summary: 日本語 160-240 字、見出しと重複しない要点（2026-06-04 倍量化）
  - summary_points: 日本語箇条書き 3-5 件、各 20-50 字
  - rationale: {importance, impact, buzz} の 3 軸、各 40-80 字
  - score: 0-100 整数 (news 性)
  - importance: high / mid / low
  - event_type: release / funding / pricing / ma / shutdown / incident / benchmark / regulation のいずれか

response_schema (JSON mode に渡す)

下記は tools/schema.py の _validate_event_extras と完全一致させる。schema 定義を schema.py に export して llm_gemini.py で再利用する。

{
  "type": "object",
  "required": ["summary", "summary_points", "rationale", "score", "importance", "event_type"],
  "properties": {
    "summary": {"type": "string", "minLength": 20, "maxLength": 280},
    "summary_points": {
      "type": "array", "minItems": 3, "maxItems": 5,
      "items": {"type": "string", "minLength": 5}
    },
    "rationale": {
      "type": "object",
      "required": ["importance", "impact", "buzz"],
      "properties": {
        "importance": {"type": "string"},
        "impact": {"type": "string"},
        "buzz": {"type": "string"}
      }
    },
    "score": {"type": "integer", "minimum": 0, "maximum": 100},
    "importance": {"type": "string", "enum": ["high", "mid", "low"]},
    "event_type": {
      "type": "string",
      "enum": ["release","funding","pricing","ma","shutdown","incident","benchmark","regulation"]
    }
  }
}

レート制御

実装は rate_limiter.TokenBucket(rate=12/60) で acquire() ブロック。time.monotonic() ベース。95 件 = 95 req → 1 req/5s → 約 8 分 + trafilatura 5 分 = 合計 13〜15 分。

失敗時の挙動

失敗パターン挙動
429 (rate exceeded)指数バックオフ 5s → 15s → 45s、3 回目で諦めて当該記事ドロップ
5xx (server error)同上
JSON schema 違反「差分を厳密 JSON で出し直して」の 1 回再投げ。2 回連続失敗で当該記事ドロップ
本文取得失敗 (trafilatura 短文・タイムアウト)当該記事ドロップ。RSS description でのフォールバックはしない (品質優先)
API キー無効・SDK エラー全体停止、run_daily.py を非ゼロ終了で Task Scheduler ログに記録

コスト・トークン見積

  • 入力トークン: 本文 2000 字 ≒ 3000 tok + system/instruction 500 tok → 約 3500 tok/req × 95 ≒ 333k tok/日
  • 出力トークン: summary + 4 points + rationale 3 軸 ≒ 350 tok × 95 ≒ 33k tok/日
  • 所要時間: Gemini 12 RPM で約 8 分 + trafilatura 平均 3 秒 × 95 ≒ 5 分 = 合計 13〜15 分
  • Free Tier 内で完結。RPM/RPD/TPM すべて余裕。クレジットカード登録不要

パラメータ

パイプラインの主要パラメータをスライダーで触り、所要時間とトークン消費のトレードオフを見る。最後に「プロンプトとしてコピー」で次セッションへ送れる。

12 rpm

15 で攻めると 429 を踏みやすい。12 が安全マージン 20%。

12 sec

trafilatura が publisher サーバーを待つ最大秒数。長すぎると 1 件で全体が詰まる。

3000 chars

Gemini に渡す本文の最大文字数。長いと入力トークン増で TPM に近づく。要約品質は 2000-3000 で頭打ち。

200 chars

本文がこれより短ければドロップ。paywall / 404 ページを食わせない閾値。

50

store.ingest_events でこの score 未満は skipped_score として捨てる。手製は 68-85、現 RSS は 50-65。

所要時間: 約 8 分 (Gemini 12 RPM × 95 件) + trafilatura 5 分 = 合計 13-15 分
